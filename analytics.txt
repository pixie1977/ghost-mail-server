ghost-mail-server
Сервер для безопасной и полностью анонимной переписки

Цель: Разработка сервера, обеспечивающего приём и передачу сообщений между клиентами с соблюдением полной приватности.

Технологии:
- Python
- FastAPI / Uvicorn
- PyJWT
- Pydantic
- REST API

Конфигурация:
- Загружается из файла .env
- Соблюдение best practices в Python

Валидация данных:
- Все входящие данные проверяются с помощью Pydantic

Криптография:
- При запуске сервера генерируется RSA-ключевая пара (4096 бит), хранящаяся исключительно в памяти
- При перезапуске сервера ключи генерируются заново

API Методы:

1. GET /get_server_key
   - Возвращает текущий открытый ключ сервера (RSA 4096 бит)
   - Запрещено кэширование ответа
   - Заголовки: Cache-Control: no-cache, no-store, must-revalidate

2. POST /register
   - Принимает зашифрованные данные регистрации (в формате JSON), зашифрованные открытым ключом сервера
   - Защита от DDOS реализована через rate limiting (максимум 5 запросов в минуту с одного IP)
   - Защита от дублирования и спуфинга через проверку уникальности логина
   - При успешной регистрации:
     - Создаётся уникальный идентификатор пользователя (на время сессии)
     - Пользователь сохраняется в файле users.json
   - Поля пользователя (все обязательны, непустые):
     - username: str
     - alias: str
     - hashed_password: str (хеширование с использованием bcrypt)
     - role: str
     - id: str (UUID сессии)
     - public: str (открытый ключ RSA 4096 бит)
   - Возвращаются соответствующие коды ошибок при несоответствии типов, длины или существовании логина
   - HTTP статусы: 201 при успехе, 400 при ошибках валидации, 409 при конфликте (логин занят), 429 при превышении лимита запросов

3. POST /login
   - Принимает зашифрованное сообщение
   - Защита от DDOS через rate limiting (максимум 10 попыток входа в минуту)
   - Расшифровывается приватным ключом сервера
   - Ожидается валидный JSON с полями:
     - username: str (до 70 символов)
     - password: str
     - public_key: str (опционально; если отсутствует — используется предыдущий; если нет вообще — ошибка)
   - Проверка пароля: сравнение хеша с использованием bcrypt
   - При успешной авторизации:
     - Генерируется JWT-токен (алгоритм из .env, по умолчанию HS256)
     - Время жизни токена: 3600 секунд (настраивается через .env)
     - Механизм обновления токена: отсутствует (требуется повторный вход)
     - Токен содержит:
       - subject: идентификатор пользователя
       - alias: отображаемое имя
       - role: роль
       - login: логин (только в режиме отладки)
     - Обновляется запись пользователя в хранилище (дата последнего входа, активность)
   - HTTP статусы: 200 при успехе, 400 при ошибках валидации, 401 при неверных учетных данных, 429 при превышении лимита запросов

4. GET /long_polling
   - Доступен только после аутентификации (требуется валидный JWT токен)
   - Ожидает новые сообщения в течение 30 секунд (long polling)
   - При отсутствии — возвращает {"status": "no new messages"}
   - Ответ шифруется открытым ключом пользователя
   - HTTP статусы: 200 при получении данных или отсутствии сообщений, 401 при отсутствии аутентификации, 400 при ошибках

5. GET /get_users
   - Возвращает список пользователей
   - Доступен только после аутентификации (требуется валидный JWT токен)
   - Без кэширования (Cache-Control: no-cache)
   - Ответ шифруется открытым ключом пользователя
   - Формат: [{"login": "login", "alias": "alias", "id": "session_id"}]
   - Если пользователь неактивен, id = null
   - Пока возвращает всех пользователей
   - TODO: добавить фильтрацию по контактам
   - HTTP статусы: 200 при успехе, 401 при отсутствии аутентификации

6. POST /get_publics
   - Принимает массив логинов
   - Возвращает список публичных ключей
   - Доступен после аутентификации (требуется валидный JWT токен)
   - Формат ответа: [{"username": "login", "publickey": "publickey"}]
   - HTTP статусы: 200 при успехе, 401 при отсутствии аутентификации, 400 при ошибках валидации

7. POST /send_message
   - Принимает JSON:
     {
       "to": "user_login",
       "message": "encrypted_message"
     }
   - Доступен только после аутентификации (требуется валидный JWT токен)
   - Проверка существования получателя
   - Сообщение сохраняется в очереди сообщений
   - HTTP статусы: 200 при успехе, 401 при отсутствии аутентификации, 400 при ошибках валидации, 404 при отсутствии получателя

Хранение данных:
- users.json: JSON-массив пользователей с блокировкой при записи (thread-safe)
- messages_queue: временная очередь сообщений (в памяти)

Механизм аутентификации:
- Все защищённые методы требуют JWT токен в заголовке Authorization: Bearer <token>
- Проверка валидности токена и срока действия

Примечания:
- Обеспечение безопасности и анонимности на всех этапах
- Поддержка долгих опросов (long polling) для получения сообщений
- Планируется расширение функционала фильтрации пользователей
